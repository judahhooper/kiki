## How do I represent a chess board in memory?

Looks like an implementation of bitboards, which I don't really understand. 
I get them in principle. 

So each piece has its position represented by a bitboard.  There are 12 piece types, and so 
a game can be represented by 12 bitboards, with optional extras associated with the answers to questions, 
such as "where are all the white pawns?" = one bitboard containing the information encoding the location
via bitboard. Additional bitboards can be cached and then recomputed depending on game state.

##How do I refer to a square?

By it's index (0-63). (x,y) correspond to (file, rank), so there's a nice trick that means that the index 
satisfies these equations:

rank = index // 8
file = index modulo 8

Each square corresponds to one of the bits on each of the instantiated bitboards, but importantly the mapping
is fixed, so there are really potentially minimum 12 sources of information (I expect part of the trick
will be knowing which bitboard to query for fastest results)


##How do I represent a piece?

The piece emerges as a property of the state of the game as represented by bitboards. Having a bitboard for
each piece type is actually enough info. Behaviour of a piece can be implemented as method on a class Position.


##How do I know whose turn it is?

Specifically for knowing who's turn is next, it's not strictly necessary to keep a turn counter. So it's
sufficient to enumerate over the colours, store that in the position and update it each move. 

A move counter would be good for time control rule enforcement or game notation, but really shouldn't be used 
to derive who's turn it is. that's quite inefficient!
